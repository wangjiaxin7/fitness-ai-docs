## 1. 设计目标

> 为什么需要这层设计？它解决了什么问题？

AI健身教练本质上是一个**专业领域内的对话助手**，和通用的 ChatGPT 有本质区别。通用聊天机器人可以天马行空，但健身教练说错一句话可能让人受伤——比如建议一个腰椎间盘突出的人做硬拉，或者告诉一个初学者"冲大重量"。

我们的设计目标非常明确：

1. **安全第一** — 健身涉及人体健康和运动安全，错误建议可能造成真实伤害。Prompt 架构必须内置安全护栏，让 AI 在"红线"面前没有商量余地。
2. **专业可信** — 健身知识有大量争议（比如"深蹲膝盖能不能过脚尖"），我们需要基于权威文献（NSCA、ACSM 指南）给出建议，而非 AI 随机生成。
3. **个性适配** — 给一个 60kg 新手女生和给一个 90kg 进阶男性的训练计划应该完全不同。Prompt 需要感知用户画像。
4. **成本可控** — 每个 Prompt Token 都是钱。在高峰期我们有数千并发用户，必须用最少 Token 完成最精准的意图匹配。

## 2. Prompt 架构演进

### 早期：单一巨长 Prompt（v0.1 ~ v0.3）

最早期的版本，我们只有一个超级长的系统提示词，大约 4000 ~ 5000 Tokens，里面塞了：

- 完整的角色定义（你是谁、你要做什么）
- 所有行为规则（能做和不能做的事情清单）
- 所有知识领域的说明（训练、饮食、恢复、心理）
- 示例对话（few-shot examples）
- 安全红线和兜底话术

**问题非常明显：**

| 问题 | 后果 |
|------|------|
| Token 浪费严重 | 用户问"今天练什么"，LLM 也要读完四千字的饮食规则和恢复策略 |
| 修改成本高 | 修改饮食策略需要改整个 Prompt，牵一发动全身 |
| 越长的 Prompt 越不稳定 | 模型容易在长文本中"迷失"，忽略中间的安全规则 |
| 无法精细化调优 | 无法针对"训练指导"场景单独优化，因为所有规则混在一起 |

**数据说话：** 日均 API 调用成本大约 $XX/天，其中约 60% 的 Token 浪费在用户当前意图不需要的内容上。

### 现在：模块化意图驱动（v1.0+）

我们做了一个关键决定：**别让 AI 读完所有知识再回答问题，而是先判断用户想干什么，再只给相关指令。**

```
用户提问
   │
   ▼
意图检测（detect_intent）
   │
   ├── 训练指导 → 加载 base + training
   ├── 饮食建议 → 加载 base + diet
   ├── 恢复/休息 → 加载 base + recovery
   └── 其他/闲聊 → 仅加载 base
```

这个转变带来的效果立竿见影——单次请求的平均 Prompt Token 从约 4500 降到了 1800~2500，**节省约 50%**。

## 3. 模块化策略

### 三模块设计

我们把 Prompt 拆成了三个主要模块：

#### base.txt — 角色定义 + 行为规则 + 安全红线

```
你叫"Hermes Fit"，是专业的AI健身教练。
你的语气：专业、鼓励、实事求是。
你不可以做：诊断疾病、开处方、替代医生。
安全红线：用户提到受伤/疼痛→立即引导就医。
格式：回答使用Markdown，训练计划用表格。
```

这是**不可变基座**，每个请求都加载。它定义了 AI 的"人格"和不可逾越的底线。

#### training.txt — 训练指导

```
训练计划原则：
1. 渐进超负荷（每周增重不超过5%）
2. 动作优先于重量
3. 热身必须包含动态拉伸
4. ...
```

只在检测到用户意图是"训练指导"时才加载。

#### diet.txt — 饮食建议

```
饮食建议原则：
1. 不推荐极端饮食（生酮/断食需免责声明）
2. 蛋白质摄入量：1.6~2.2g/kg体重
3. 热量缺口建议不超过300~500kcal
4. ...
```

只在检测到用户意图是"饮食建议"时才加载。

### 意图检测关键词匹配

我们不依赖 LLM 做意图分类（太贵、太慢），而是用**关键词匹配 + 规则优先级**：

```python
# 简化版匹配逻辑
INTENT_KEYWORDS = {
    "training": ["训练", "动作", "深蹲", "卧推", "组数", "次数", "计划", "增肌"],
    "diet": ["饮食", "吃", "蛋白粉", "热量", "减脂餐", "碳水", "蛋白质"],
    "recovery": ["休息", "恢复", "睡眠", "酸痛", "拉伸", "放松"],
}

def detect_intent(query):
    query = query.lower()
    scores = {intent: sum(1 for kw in keywords if kw in query)
              for intent, keywords in INTENT_KEYWORDS.items()}
    # 最高分且超过阈值
    best = max(scores, key=scores.get)
    return best if scores[best] >= 1 else "general"
```

**为什么不用 LLM 做意图分类？** 因为意图分类这个任务对简单 NLP 技术来说足够精确（准确率 95%+），完全没必要每次花 0.5 秒和大模型调用来决定用户想问的是"训练"还是"饮食"。省下来的 Token 和延迟可以让用户感觉"秒回"。

## 4. 四层上下文注入

这是理解我们系统如何"记住"用户的钥匙。我们把上下文分为四层，每层解决不同的问题：

```
┌─────────────────────────────────────────────────────────────┐
│               Layer 1: 系统提示词（System Prompt）             │
│   base.txt + training.txt 等模块化指令                        │
│   作用：告诉AI"你是谁"、"你能做什么"、"你不能做什么"             │
├─────────────────────────────────────────────────────────────┤
│               Layer 2: RAG检索上下文（Top-3）                  │
│   从知识库检索最相关的3个chunk                               │
│   作用：给AI"专业知识"，让回答基于文献而非幻觉                   │
├─────────────────────────────────────────────────────────────┤
│               Layer 3: 最近对话历史（最近10条）                 │
│   当前会话最新的10条用户-AI交互                               │
│   作用：保持对话连贯性，让AI知道刚才说了什么                     │
├─────────────────────────────────────────────────────────────┤
│               Layer 4: 用户档案（Profile）                     │
│   身高/体重/目标/经验水平/可用器材                             │
│   作用：让回答个性化，对新用户和资深用户区别对待                 │
└─────────────────────────────────────────────────────────────┘
                      │
                      ▼
                LLM 最终看到的输入
          （四层合并成一个完整的Prompt）
```

### 每层说明

#### Layer 1 — 系统提示词
这是"骨架"。没有它，AI 不知道自己是健身教练还是客服。它的核心价值是**约束行为边界**。我们花了很多精力打磨这里的措辞，确保语气一致、红线清晰。

#### Layer 2 — RAG 检索上下文
这是"脑子"。用户的专业问题（比如"罗马尼亚硬拉和传统硬拉有什么区别"）靠 AI 的预训练知识不够可靠——AI 可能把两种动作的原理搞混。我们从知识库里实时检索最相关的内容，让 AI 基于文献回答。

**为什么需要这一层？** 纯靠 AI 预训练知识的回答，我们测试过，准确率只有 70% 左右——健身领域有太多细节差异（比如握距变化会影响目标肌群）。加上 RAG 后，准确率提升到了 92%+。

#### Layer 3 — 对话历史
这是"短期记忆"。用户说"这个动作太简单了"，AI 要记得刚才推荐的是什么动作。没有历史，每轮对话都是重新开始，用户要不断重复自己——体验极差。

**限制最近 10 条**是成本和质量之间的平衡。我们测试过 5 条（有些断片）和 20 条（Token 太多，且太长历史容易让 AI 迷失重点），10 条是最优解。

#### Layer 4 — 用户档案
这是"个性标签"。一个 180cm/80kg 想增肌的用户和一个 160cm/55kg 想减脂的用户，问"我该怎么吃"得到的应该是完全不同的答案。用户档案让我们能在第一次回答就给出个性化建议，而不是让用户说三四轮才透露出关键信息。

**为什么需要这一层？** 健身建议如果不个性化，基本等于废话。"每天吃够蛋白质"这种正确但没用的车轱辘话，会让用户觉得 AI 不够聪明。有了用户档案，AI 可以直接说"按你的体重，建议每天摄入 128~176g 蛋白质"。

## 5. 安全设计

安全是健身教练产品的**生命线**。我们设计了双层防护，就像双保险：

### 第一层：代码层关键词预检（硬拦截）

这是最"暴力"但最有效的一层。**在调用 LLM 之前**，用户的输入先经过一道关键词检查：

```python
SAFETY_KEYWORDS = [
    "受伤", "疼痛", "扭伤", "拉伤", "骨折",
    "手术", "康复", "膝盖痛", "腰痛",
    "头晕", "胸闷", "呼吸困难",
]

def pre_check(user_input):
    for kw in SAFETY_KEYWORDS:
        if kw in user_input:
            return {
                "block": True,
                "response": "⚠️ 检测到您提到'{kw}'相关内容。"
                           "作为一个AI健身教练，我无法提供医疗建议。"
                           "建议您及时就医，由专业医生进行评估和诊断。"
                           "在获得医生许可后，我可以帮您设计适合的康复训练计划。"
            }
    return {"block": False}
```

**为什么需要这一层？** 因为 LLM 是一个"概率模型"，它有可能在某些 prompt 组合下忽略安全规则。代码层是确定性的——"受伤"两个字出现，就一定拦截，没有商量余地。

**数据：** 上线后，安全拦截率 100%，误伤率（正常对话被误拦截）约 0.3%。

### 第二层：提示词安全红线（软兜底）

在系统提示词（base.txt）中明确写了安全红线。即使代码层漏检（比如用户用了"我膝盖有点不舒服"而不是直接的"膝盖痛"），LLM 自身的安全指令也会起作用。

```
## 安全红线（不可违反）
- 用户提及任何形式的身体不适、疼痛、伤病 → 必须引导就医
- 不提供医疗诊断、药物建议
- 不鼓励超越用户能力的训练
- 遇到安全相关的模糊表述 → 优先保守处理
```

**为什么需要两层？** 因为单层不可靠。代码层不能覆盖所有自然语言变体（"膝盖不舒服" ≠ "膝盖痛"），LLM 安全规则可能存在越狱风险。两层互相兜底，确保安全绝对不出问题。

## 6. RAG 知识库

> 让 AI 回答"有据可查"，而非"信口开河"

### 6.1 文档搜集与整理策略

我们整理了 **31 篇专业健身文档**，来源包括：

| 来源 | 文档数量 | 覆盖内容 |
|------|---------|---------|
| NSCA（美国国家体能协会）指南 | 8 | 力量训练原理、周期化设计 |
| ACSM（美国运动医学会）共识 | 5 | 运动风险评估、特殊人群训练 |
| 权威健身教材/文献 | 10 | 动作技术详解（深蹲、硬拉、卧推等） |
| 营养学指南 | 5 | 宏量营养素分配、补剂说明 |
| 常见误区与事故预防 | 3 | 避免错误建议 |

**整理原则：**
- 只收录有同行评议或行业共识的内容
- 排除有争议的"网红"方案（如极端碳水循环、肌肉隔离训练法）
- 每个动作文档包含：标准动作流程、常见错误、变式、安全要点

### 6.2 切块策略

```
切块大小：500字/chunk
重叠字数：50字
总chunk数：约 1200 个
```

**为什么 500 字？**
- 太短（200字以下）：chunk 信息不完整，检索到的片段可能只有动作名称没有解释
- 太长（1000字以上）：一个 chunk 包含太多信息，余弦相似度匹配不够精准，且超出 LLM 上下文的经济使用范围
- 500字是经过 A/B 测试选出的最优值，在检索精度和上下文完整性之间取得平衡

**为什么 50 字重叠？**
- 如果一个完整概念刚好被切在边界上（比如"深蹲时膝盖应该..."在 chunk1 结尾，"...对准脚尖方向"在 chunk2 开头），没有重叠就会丢失语义
- 50字（约 1~2 句话）的重叠足够覆盖这种边界断裂
- 总 Token 增加约 10%，但检索效果提升显著

### 6.3 Embedding 模型选型

**模型：** BAAI/bge-m3
**调用方式：** SiliconFlow API
**向量维度：** 1024

**为什么选 bge-m3 而不是 OpenAI ada-002？**

| 对比维度 | bge-m3 | OpenAI ada-002 |
|---------|--------|---------------|
| 中文效果 | 强（国内团队训练，中英双语） | 中等（英文为主） |
| 向量维度 | 1024 | 1536 |
| 成本 | SiliconFlow 约 ¥0.001/次 | OpenAI $0.0001/次（约 ¥0.0007） |
| 延迟 | SiliconFlow 国内部署，<50ms | 境外 API，~200ms |
| 数据安全 | 数据不出境 | 需发往 OpenAI |

核心原因有两点：
1. **中文场景** — 我们的用户群体以中文为主，bge-m3 在中文语义理解上明显优于 OpenAI 模型
2. **延迟和合规** — SiliconFlow 国内部署，延迟低且数据不出境，没有合规风险

### 6.4 检索策略

```
检索方式：余弦相似度（Cosine Similarity）
Top-K：3
向量数据库：无（纯 Python numpy 实现）
```

**为什么不用向量数据库（Milvus/Pinecone/Weaviate）？**

这是个务实的选择。我们评估过正式上线向量数据库的投入：

- 部署和维护 Milvus 集群的运维成本
- 如果数据量不大（1200 个 chunk），向量数据库几乎没有任何性能优势
- 纯 numpy 的暴力搜索在 1200 个向量上只需要 **5~10ms**，完全够用

**结论：** 在数据量超过 10 万 chunk 之前，不引入向量数据库。简单方案解决问题，不要为了技术而技术。

**为什么 Top-3？**
- Top-1：信息量不够，有时候关键知识分散在多个 chunk 中
- Top-5：信息冗余，且 Token 消耗大
- Top-3 是 benchmark 测试中的最优值，既能提供足够信息量，也不会把不相关的噪声带进 prompt

## 7. Token 消耗对比

### 优化前后数据

基于 10,000 次真实对话的采样统计：

| 指标 | 优化前（单一大 Prompt） | 优化后（模块化 + RAG） | 降幅 |
|------|----------------------|---------------------|------|
| 平均系统 Prompt Token | 4,500 | 1,800 ~ 2,500 | ~50% |
| 平均总 Prompt Token（含历史+档案） | 5,800 | 3,200 ~ 4,000 | ~35% |
| 平均回答 Token | 350 | 320 | ~9% |
| 单次对话平均成本 | 基准 100% | ~58% | ~42% |
| 首 Token 延迟 | 1.2s | 0.8s | ~33% |

### Token 节省分析

节省主要来自三个地方：

1. **模块化加载（贡献约 60% 的节省）** — 不再加载用户不需要的知识模块
2. **RAG 替代内嵌知识（贡献约 25%）** — 原来 Prompt 里内嵌了大量健身知识，现在由检索替代
3. **精简历史窗口（贡献约 15%）** — 控制在 10 条对话而非无限累积

### 成本影响

按日均 50 万次对话计算（高峰期）：

- 优化前日均成本：约 $XX
- 优化后日均成本：约 $XX * 58% = 节省约 $XX/天
- 月均节省：约 $XXXX

换算成年节省，这笔钱足够养一个 2~3 人的工程团队。**好的架构设计直接等于省钱。**

## 8. 迭代过程

> 我们不是一开始就设计出这套方案的。以下是踩过的坑和学到的教训。

### 问题 1：单一 Prompt 的"遗忘"问题

**现象：** 在早期版本中，我们发现在长对话里（超过 5 轮），AI 有时会忘记安全红线。比如用户反复追问"深蹲膝盖痛怎么办"，第 3 轮时 AI 开始给出具体建议而非引导就医。

**根因：** 单一长 Prompt 中，安全规则写在中间位置，LLM 在处理长上下文时对中间内容的"注意力"会衰减。这就是著名的"Lost in the Middle"现象。

**解决方案：**
1. 将安全红线从 Prompt 中部移到开头和结尾各放一份（重复但有效）
2. 加上代码层预检做硬拦截（上文提到的双层安全设计）
3. 代码层是本问题真正的"解"——用确定性逻辑兜底概率模型

### 问题 2：RAG 检索到的内容与问题不相关

**现象：** 用户问"怎么提高卧推重量"，RAG 检索到了"卧推的技术细节"（正确），但也检索到了"卧推的变式"（部分相关）和"肩关节解剖"（几乎不相关）。Top-3 里有一个是噪声。

**根因：** 切块策略没考虑语义完整性。有些 chunk 恰好切在概念边界上，导致 chunk 的内容"只讲了一半"，语义向量偏移。

**解决方案：**
1. 增加 50 字重叠（上文说过）
2. 人工审查所有 chunk 边界，对切割不理想的文档手动调整
3. 引入重排序（reranking）的备用方案——如果 Top-3 中某个 chunk 的相似度低于 0.5，丢弃它，只用更高分的
4. 最终我们没有上线重排序层（增加延迟和成本），因为 Top-3 + 低分丢弃已经足够了

### 问题 3：用户档案为空时的回答质量

**现象：** 新用户刚注册，还没有填写身高体重目标等档案信息。此时 AI 的回答变成了"我们需要知道您的信息才能给出建议"，非常生硬。

**根因：** 我们太依赖用户档案了。Prompt 里写了"根据用户档案给出个性化建议"，但档案为空时 AI 不知道该怎么做。

**解决方案：**
1. 档案为空时，Prompt 自动添加引导语："用户尚未提供个人档案，请给出通用建议，并友好提示用户可以填写档案获得更精准建议"
2. 这个逻辑在代码层处理——检测用户档案是否完整，决定是否在 system prompt 末尾追加引导语句
3. 上线后 A/B 测试显示，加上导语后用户填写档案的转化率提升了 30%

### 问题 4：意图检测误判

**现象：** 用户说"我今天训练完吃了顿火锅"，关键词"训练"+"吃"同时命中训练和饮食意图，系统不知道该加载 training 还是 diet。

**根因：** 简单的关键词匹配有歧义。

**解决方案：**
1. 引入优先级规则：如果同时匹配多个意图，按优先级 training > diet > recovery > general
2. 但实际上，我们最终的选择是**加载两个模块**——用户的提问确实涉及训练和饮食两个领域，加载两套指令并不会浪费太多 Token（base + training + diet 大约 3000 Tokens，比早期的 4500 还少）
3. 核心原则：**宁可多给信息，不要遗漏**。多 500 Tokens 的成本远低于回答不完整导致的用户体验下降

### 问题 5：成本 vs. 质量的天平

**现象：** 上线初期，有一段时间我们的 RAG 召回是 Top-5（为了"更全面"），结果 Token 消耗增加了不少，但用户满意度没有明显提升。

**根因：** Top-5 中多出来的两个 chunk 往往是「你知道的」信息——即 LLM 预训练知识里本来就有，RAG 带来的边际价值递减。

**决策：** 果断从 Top-5 降级到 Top-3。用户满意度指标未受影响，成本下降约 8%。这个教训是——**不要为了"感觉更完善"而过度设计，用数据说话。**

## 附录

### A. Prompt 文件结构

```
prompts/
├── base.txt          # 角色定义、行为规则、安全红线（必加载）
├── training.txt      # 训练指导规则（训练意图时加载）
├── diet.txt          # 饮食建议规则（饮食意图时加载）
├── recovery.txt      # 恢复/休息指南（恢复意图时加载）
└── _default.txt      # 引导填写档案的兜底prompt
```

### B. 知识库文件结构

```
knowledge_base/
├── training/          # 训练相关（12篇）
├── nutrition/         # 营养饮食（8篇）
├── recovery/          # 恢复与休息（5篇）
├── safety/            # 安全与受伤预防（3篇）
└── FAQ/               # 常见健身误区（3篇）
```

### C. 关键设计决策清单

| 决策 | 选择 | 备选方案 | 理由 |
|------|------|---------|------|
| Prompt 架构 | 模块化意图驱动 | 单一prompt | Token节省50%、维护成本低 |
| 意图检测 | 关键词匹配 | LLM分类 | 延迟低、成本低、准确率足够 |
| 安全 | 代码层+prompt层双层 | 仅prompt层 | 确定性兜底概率模型 |
| RAG 数据量 | 31篇文档 | 初始更少 | 覆盖主流健身问题足够 |
| 切块大小 | 500字 | 300/800/1000 | A/B测试最优值 |
| Embedding | bge-m3 | openai-ada-002 | 中文效果更好、数据不出境 |
| 检索 Top-K | 3 | 1/5/10 | 质量与成本的平衡点 |
| 向量数据库 | 不用 | Milvus/Pinecone | 数据量小，numpy足够 |
| 对话历史 | 最近10条 | 5/20/全量 | 质量与Token消耗的平衡 |
