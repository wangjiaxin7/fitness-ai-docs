# 技术选型决策 — Dify vs 自研

## 背景

AI 健身教练的 AI 对话模块最初使用 Dify 低代码平台搭建，后切换为直调 DeepSeek API + 自建 RAG。本文记录决策过程和取舍逻辑。

---

## 1. 为什么最初选 Dify

### 1.1 选型时的考量

| 考量 | Dify 的优势 |
|------|------------|
| 开发速度 | 可视化编排 Prompt，几分钟就能跑通一个对话流 |
| 门槛低 | 不需要写后端代码，拖拽配置即可 |
| RAG 内置 | 上传文档就能用，不用自己实现向量检索 |
| 模型切换 | 支持多个 LLM 供应商，切换只需改配置 |

### 1.2 初期验证效果

用 Dify 快速验证了：
- Prompt 设计是否有效（角色设定 + 健身知识注入）
- 用户对 AI 对话的接受度
- 基础的 RAG 检索效果

**结论**：Dify 是很好的原型验证工具，快速证明了产品方向可行。

---

## 2. 遇到了什么问题

### 2.1 平台 Bug：续对话失效

**现象**：Dify 的对话 API 在多轮对话时，偶尔丢失上下文，导致 AI "失忆"。

**影响**：用户体验断裂——"我刚才说的它全忘了"。

**排查过程**：
- 检查 conversation_id 传递是否正确 → 正确
- 检查 Dify 日志 → 发现 Dify 内部偶发的 conversation 状态丢失
- 向 Dify 社区反馈 → 确认是已知 Bug，暂无修复时间表

**结论**：依赖第三方平台意味着依赖它的 Bug 修复节奏。

### 2.2 RAG 白名单限制

**现象**：Dify 的 RAG 功能对 embedding 模型有白名单限制，不支持自选模型。

**影响**：
- 无法使用更适合中文的 embedding 模型（如 BAAI/bge-m3）
- 检索质量受限于平台提供的默认模型
- 无法调整 chunk 策略（chunk size、overlap）

**结论**：RAG 是本产品的核心能力，必须有完全的控制权。

### 2.3 调试困难（黑盒）

**现象**：Dify 的对话流是黑盒，出了问题很难定位。

**具体表现**：
- 无法看到实际发送给 LLM 的完整 Prompt
- 无法追踪 RAG 检索的具体结果
- 无法自定义重试逻辑和错误处理
- 日志不够详细，无法定位性能瓶颈

**结论**：产品逻辑变复杂后，黑盒调试的成本远高于自研。

### 2.4 流式响应兼容性

**现象**：Dify 的 SSE 流式响应格式与标准 OpenAI 格式不完全兼容。

**影响**：前端需要写额外的适配层，增加了前端复杂度。

---

## 3. 切换决策

### 3.1 决策框架

| 维度 | Dify | 自研 | 权重 |
|------|------|------|------|
| 开发速度（初期） | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | 高 → 但只在初期重要 |
| 控制力 | ⭐⭐ | ⭐⭐⭐⭐⭐ | 高 → 随产品复杂度增长 |
| 调试效率 | ⭐⭐ | ⭐⭐⭐⭐⭐ | 高 → 影响迭代速度 |
| RAG 定制能力 | ⭐⭐ | ⭐⭐⭐⭐⭐ | 高 → 核心功能 |
| 部署复杂度 | ⭐⭐⭐ | ⭐⭐⭐⭐ | 中 → Dify 需要额外服务 |
| 维护成本 | ⭐⭐⭐ | ⭐⭐⭐ | 中 → 差不多 |

### 3.2 核心结论

> **Dify 降低了入门门槛，但降低了系统控制力。产品逻辑变复杂后，控制力比入门速度更重要。**

这是一个**阶段性的正确选择**：
- **阶段 1（验证期）**：用 Dify 快速验证产品方向 → ✅ 正确
- **阶段 2（开发期）**：切换为自研方案，获得完全控制权 → ✅ 正确

## 4. 自研方案设计

### 4.1 AI 对话模块

**方案**：直调 DeepSeek API（OpenAI 兼容格式）

```python
# 核心调用逻辑
response = requests.post(
    "https://api.deepseek.com/v1/chat/completions",
    headers={"Authorization": f"Bearer {api_key}"},
    json={
        "model": "deepseek-chat",
        "messages": [
            {"role": "system", "content": system_prompt},
            *conversation_history[-10:],  # 滑动窗口 10 条
            {"role": "user", "content": user_message}
        ],
        "stream": True  # 流式响应
    }
)
```

**优势**：
- 完全控制 Prompt 构建过程
- 可以自定义重试、超时、错误处理
- 流式响应格式标准，前端适配简单

### 4.2 RAG 模块

**方案**：自建 Python RAG（SiliconFlow BAAI/bge-m3 + 余弦相似度）

```
文档 → 切块（500字/chunk，50字重叠）
  ↓
SiliconFlow Embedding API → 向量化
  ↓
本地 JSON 存储向量索引
  ↓
用户查询 → 向量化 → 余弦相似度 → Top-3
```

**优势**：
- 完全控制 chunk 策略（size、overlap）
- 可以自由选择 embedding 模型
- 不依赖向量数据库，部署简单
- 31 篇文档的规模，纯 Python 足够

**为什么不引入向量数据库**：
- 31 篇文档 ≈ 242 个 chunks，规模太小
- 引入 Milvus/Pinecone 增加部署复杂度
- 纯 Python + JSON 索引足够，性能不是瓶颈

### 4.3 安全模块

**方案**：双层防护

```
用户输入
  ↓
第 1 层：代码关键词预检（确定性）
  - 伤病关键词列表匹配
  - 命中 → 直接返回就医建议，不调 LLM
  ↓ 未命中
第 2 层：提示词安全红线（概率性）
  - System Prompt 中定义安全规则
  - LLM 自行判断是否涉及伤病
```

**为什么不只用提示词**：
- 提示词是概率性的，LLM 可能忽略
- 伤病问题的代价太高（用户受伤），必须有确定性拦截
- 代码层预检是 100% 确定的，作为兜底

## 5. 切换过程

### 5.1 迁移步骤

1. 保留 Dify 的 Prompt 设计，提取为代码中的常量
2. 实现 DeepSeek API 调用模块（OpenAI 兼容格式）
3. 实现 RAG 模块（embedding + 余弦相似度）
4. 实现安全预检模块
5. 前端适配新的 API 格式
6. 并行运行 1 周，对比效果
7. 下线 Dify

### 5.2 迁移风险与应对

| 风险 | 应对 |
|------|------|
| 自研质量不如 Dify | 并行运行对比，数据说话 |
| 迁移期间服务中断 | 新旧方案并行，无缝切换 |
| RAG 检索质量下降 | 用相同 query 对比 Top-3 结果 |

### 5.3 迁移结果

| 指标 | Dify 方案 | 自研方案 | 变化 |
|------|----------|---------|------|
| 首 token 延迟 | 2-5 秒 | 1-3 秒 | ↓ 降低 |
| RAG 检索相关性 | ~70% | ~85% | ↑ 提升 |
| 多轮对话稳定性 | 偶发丢失上下文 | 稳定 | ↑ 显著改善 |
| 调试效率 | 黑盒，难定位 | 完全日志，快速定位 | ↑ 显著改善 |
| 部署复杂度 | Dify + PostgreSQL + Redis + Nginx | Flask + PostgreSQL | ↓ 简化 |

---

## 6. 总结

### 6.1 决策原则

> **快速验证用平台，产品复杂后切自研。**

这不是"推翻之前的选择"，而是"在正确的时间做正确的切换"。

### 6.2 经验教训

1. **低代码平台是很好的原型工具**，但不是生产工具
2. **控制力是产品成熟后的核心需求**，不能为了开发速度牺牲控制力
3. **黑盒调试的成本会随产品复杂度指数增长**
4. **并行对比是迁移的最安全方式**

### 6.3 适用场景判断

| 场景 | 建议 |
|------|------|
| 快速验证产品方向 | 用 Dify / 低代码平台 |
| 产品逻辑简单（单轮问答） | 用 Dify 足够 |
| 产品逻辑复杂（多轮 + RAG + 安全） | 自研 |
| 需要深度定制 RAG | 自研 |
| 需要精细的错误处理和重试 | 自研 |
